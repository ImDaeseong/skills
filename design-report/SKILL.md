## Step 1: Design direction (when the request involves a visual artifact)

Before writing any HTML/CSS/component code for a deck, landing page, or prototype:

1. **Lock scope with sharp questions, then a low-fidelity pass — before any visual polish.** Ask AskUserQuestion-style clarifying questions specific to the artifact (primary device, tone/persona, which screens actually need mockups, key user segments, accessibility requirements, **whether a light/dark mode pair is needed**) rather than guessing scope. Then produce the *lowest*-fidelity wireframe/outline that conveys the structure — not a medium-fidelity one; medium-fidelity is the expensive middle ground that gets thrown away once real feedback arrives. Only move to the visual-direction work below once the low-fidelity pass is confirmed — this locks feature/scope decisions cheaply before tokens go into high-fidelity output that scope changes would waste.
2. **Check for an existing `DESIGN.md` first** (project root, or wherever the user's other design artifacts live). If one exists, read and follow its tokens/rationale rather than deciding a new direction from scratch — a one-shot direction that isn't persisted breaks down on the second page/slide/mockup; a `DESIGN.md` is what carries it across media and tool calls.
3. If none exists, commit to a specific typography pairing, color direction, and layout approach — state the choice and the one-line reason, not a menu of options. Do not default to Inter/Roboto/Arial/Space Grotesk without a stated reason; per the `frontend-design` skill's own rule, these read as "unexamined AI defaults." **If the user has or points to reference images** (competitor screenshots, a Dribbble shot, a physical product photo for a material/tactile feel), Read them and anchor the direction to specifics actually visible in them — name what's being borrowed from each (a proportion, a corner-radius feel, a button's tactile weight) rather than only describing the direction in prose with no visual anchor. Combining more than one reference (e.g. a UI reference for layout + a physical-object reference for texture/depth) produces a more distinctive result than either alone — same blend-don't-clone principle as the mascot guidance below (item 6).
4. **Persist that decision as a `DESIGN.md` file** following the `google-labs-code/design.md` spec (see Attribution): YAML frontmatter with `name`, `colors`, `typography`, `rounded`, `spacing`, `components` tokens, plus a markdown body explaining the rationale in prose. Write it once the direction is committed, not after every artifact — later artifacts in the same session/project read this file instead of re-deriving the direction. **Check text/background color pairs against WCAG AA (4.5:1 for normal text)** before finalizing the token values — this is an explicit requirement of the spec itself (verified directly against `docs/spec.md`), not an optional nicety; a direction that fails this check needs an adjusted color token, not a footnote. **If light/dark mode was confirmed needed (Step 1):** the spec has no theming/mode-switching mechanism at all (verified — the word "dark" appears nowhere in `docs/spec.md` except inside an unrelated color-name example) — do not assume the schema handles this. Document both palettes explicitly: either a second full `colors` token block labeled by mode in the prose body, or a stated mapping convention (e.g. each color token gets a light/dark pair) — pick one and state it, since the spec itself won't tell you how.
5. **Include a motion/interaction principles subsection in the `DESIGN.md` prose body** — the spec's token schema has no motion/animation field (verified directly against `docs/spec.md`; only colors/typography/rounded/spacing/components are tokenized), but the free-form prose body is exactly where this belongs. State the transition/easing feel (e.g. "spring, slightly overshooting" vs. "linear, snappy"), which states get a loading/searching micro-interaction versus a static swap, and haptic feedback points if the target is mobile. Skipping this is the single biggest reason a design reads as "obviously AI-generated" rather than polished — static layouts with the same content don't fail on structure, they fail on feel.
6. **If the artifact needs a mascot or custom illustration**, don't generate one-off images per screen: commission or draw a single original base illustration first, then generate on-brand variants from that base (same character, different poses/states) rather than prompting a new character each time. Do not clone an existing brand's mascot/style directly — blend multiple references into something original, and say so if the direction was inspired by named references.
7. If the input is a `biz-council` design document, pull its "What Gets Built" section as the content skeleton — do not invent features not in that document.
8. **If the artifact actually renders (a live webpage/prototype, not a static mockup), verify by rendering it, not by re-reading the code.** Check what's actually available (a browser-automation tool, a screenshot capability) before assuming it exists — same discipline as Step 2.2's tool check. Code that looks correct and a page that renders correctly are different claims; a WCAG contrast check on token values (Step 4) doesn't catch a layout that only breaks at runtime.

## Step 2: Report generation (when the request is to format findings into a document)

1. Identify target format (DOCX/PPTX/XLSX/PDF) from the request.
2. Check what's actually available in this session/host (an installed Office MCP tool, a Python environment with the relevant library, or the user has one of the Attribution-table skills installed separately) — do not assume a tool exists; verify with a tool call (e.g., check for an MCP resource, or `command -v` for a CLI dependency) before promising output in that format.
3. If nothing is available, say so and ask whether to install one of the Attribution-table options (respecting the license notes above) or produce a plain Markdown report instead. **If the report is meant to be visually reviewed rather than read as plain text** (a spec, a design review, a dashboard-style summary), consider an HTML Artifact instead of a flat document — higher information density and visual structure than markdown, at the cost of more tokens to generate; offer it as an option, don't default to it silently.
4. Structure the report content around whatever it's summarizing (e.g. a `biz-council` document's six sections) — do not add sections the source material doesn't support.

## What this skill does not do

Does not run research (that's `biz-council`), does not fabricate a design or report when the underlying content/tooling isn't there — say so per LAW 0 instead of producing a plausible-looking placeholder.
